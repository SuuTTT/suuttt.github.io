Using GPT Codex to update a GitHub Pages blog

I keep using GPT Codex (the coding-specialized GPT) to edit this site. Here is the lightweight workflow that keeps it reproducible, Git-friendly, and safe for GitHub Pages.

1) Prep the repo on your machine

  • Clone or pull the blog repo locally so Codex can inspect files without network calls.
  • Open an interactive shell for the model (or share a file tree) so it can read HTML, CSS, JS, and templates.
  • Point Codex to any house rules (e.g., AGENTS.md, CONTRIBUTING notes, or style guides) so it follows local conventions.

2) Frame the prompt like a dev task

  • State the goal (“add a new post on Codex workflow”) and constraints (static HTML only, keep PaperMod layout intact, update archive & tags).
  • Ask for a plan first if the change is large; small edits can go straight to patching.
  • Request citations (file + line) in the final response so you can quickly audit the diff.

3) Edit with small, diffable steps

  • Let Codex create or update files in small chunks—short Markdown/HTML sections, scoped CSS tweaks, or single JS helpers—so the diff stays readable.
  • Prefer copy-editing existing blocks (navigation, sidebars, tag lists) instead of rebuilding them from scratch to avoid layout regressions.
  • Keep new assets static (images, scripts) so GitHub Pages can serve them without a build step.

4) Verify before committing

  • Ask Codex to summarize the changes and confirm which files were touched.
  • Skim the rendered HTML (or run a local preview) to ensure sidebars and archives still align.
  • Update tag clouds and archive lists so navigation surfaces the new content.

5) Commit and push confidently

  • Commit on a feature branch with a clear message (e.g., add codex workflow article), then open a PR with a short summary + testing notes.
  • On GitHub Pages, a simple push is enough—no extra CI build is required if you stick to static assets.

6) Safety checklist I use with Codex

  • No secrets in prompts or repo (treat the environment as public).
  • No client-side fetches to private APIs; keep interactivity self-contained.
  • Validate links and relative paths so cross-page navigation keeps working.
  • Prefer incremental commits; if something looks wrong, revert the last commit and re-prompt.

That’s it: a repeatable loop of prompt → small diff → verify → commit keeps Codex productive without breaking the blog.

Distill-style experiments on Hugo PaperMod

Distill posts are really just long-form HTML with inline CSS and a pinch of JavaScript for scrollytelling or interactive notes. You can recreate that feel on Hugo + PaperMod without any backend by leaning on three patterns:

Tiny interactive widgets via shortcodes

Wrap any small widget (like a slider + output span) in a shortcode so authors can drop {{< slider-demo >}} inside Markdown. The shortcode template can live in layouts/shortcodes/slider-demo.html and include both the markup and a minimal script:

<div class="distill-widget">
  <input type="range" min="0" max="10" value="5" oninput="this.nextElementSibling.textContent = this.value" />
  <span>5</span>
</div>
<style>
.distill-widget { display:flex; gap:0.5rem; align-items:center; }
</style>

Because the shortcode is pure HTML/CSS/JS, it survives GitHub Pages builds and keeps the Distill vibe.

Bigger experiments via iframes

For heavier sketches (Observable notebooks, three.js demos, etc.), export them as standalone static HTML files and place them under static/experiments/ in your Hugo project. Then embed with an iframe from your post body:

<figure class="distill-embed">
  <iframe src="/experiments/attention-demo/index.html" title="Attention demo" loading="lazy"></iframe>
  <figcaption>Fully static embed that still works on GitHub Pages.</figcaption>
</figure>
<style>
.distill-embed iframe { width:100%; min-height:420px; border:1px solid #e0e0e0; border-radius:8px; }
</style>

This keeps interactivity isolated and lets you iterate on the experiment without touching the main post content.

Custom layouts for special posts

When a piece needs a bespoke Distill-like layout (floating sidenotes, scrolling stepper, etc.), create a custom single layout: layouts/_default/single.distill.html. In your front matter, set layout: distill to opt in. The layout can add sticky sidenotes and include a dedicated stylesheet at assets/css/distill.css for typography tweaks.

Author workflow tips

  • Keep reusable CSS (grid helpers, note callouts) in assets/css/distill.css and import it globally via params.assets in config.yaml.
  • Namespace your JavaScript (e.g., window.distillWidgets = {...}) to avoid conflicts with PaperMod scripts.
  • Document each shortcode with a minimal usage example so future posts stay consistent.

Hierarchical clustering on KGE

TASK Overview

任务

知识库嵌入，得到三元组(h,r,t)中实体和关系的向量表示，进一步用于下游任务，比如三元组分类（triple classification），实体识别，连接预测(link prediction)，知识图谱补全。

数据集

常见的数据集FB15k, WN18, YAGO3-10及其变种分别是从freebase,wordNet, YAGO中提取的三元组。目前最流行的数据集如下[^2]：

SOTA

比较 SOTA 结果比较困难，因为存在不同的训练策略（例如负例生成策略）、不同或次优的超参数选择。

评价指标

连接预测，即预测首尾实体或是关系。这个问题的特点是没有ground-truth的负例，因核心想法是利用对嵌入进行排序来将该问题转化为Learning-To-Rank 问题：正三元组在局部封闭世界假设下相对于合成负例的排名如何。

具体有MR，MMR，hit@k

方法与实现

模型

基于Knowledge Graph Embedding with Hierarchical Relation Structure [^1]的模型：在transE模型的基础上，对嵌入进行聚类后进一步学习。transE模型就是通过生成负例，定义让正例得分高于负例的得分函数，以其为目标函数对嵌入进行学习。

现在的问题

1需要指定聚类数量k，

2 语义层次人为设定成3层，每层只是平面聚类。

3 可以进一步考虑对entity聚类
